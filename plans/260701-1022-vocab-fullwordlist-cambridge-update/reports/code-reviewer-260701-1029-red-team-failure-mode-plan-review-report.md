# Red Team Review: vocab-fullwordlist-cambridge-update plan

Role: Failure Mode Analyst / Flow Tracer. Verified against actual code paths in `js/english-generators.js`, `js/game-engine.js`, `js/vocab-data-starters.js`, `js/vocab-data-a2key.js`, `index.html`.

## Finding 1: No automated syntax check before wiring new part files — one bad string can silently take down Math too
- **Severity:** Critical
- **Location:** Phase 4, section "Implementation Steps" (steps 3-4) and "Risk Assessment" (risk 1)
- **Flaw:** The plan's only defense against a broken `.js` part file is manual "check variable names" and opening the browser afterward (Phase 4 step 4). There is no step that runs a syntax check (`node -c file.js`) on each of the ~20-30 hand-authored part files before they are added to `index.html`.
- **Failure scenario:** `index.html:191-193` currently loads `vocab-data-starters.js` and `vocab-data-a2key.js` **before** `game-engine.js`, and `game-engine.js:563,567,571` is the dispatcher used by BOTH subjects (`state.currentSubject === 'english' ? ... : <math path>`). Script tags execute synchronously and a `SyntaxError` in any one part file (e.g. an unescaped apostrophe in a Vietnamese translation like `vi: "con chuột's"`, a stray trailing comma, or a missing closing brace across 700-1000 hand-typed entries) aborts the entire remaining script pipeline. Since the new part files sit before `game-engine.js` in load order, this doesn't just disable English — `game-engine.js`, `english-generators.js`, `english-engine.js`, and `main.js` never execute at all, breaking Math too. Confirmed: `node -c` is available (`node v25.9.0` in this environment) and currently passes on both existing vocab files, so the check is trivial and free but is never specified.
- **Evidence:** `js/game-engine.js:563` `if (state.currentSubject === 'english' && typeof renderEnglishQuestion === 'function') return renderEnglishQuestion();` (same dispatcher pattern at lines 567, 571) — proves game-engine.js is shared infra for Math, not English-only. `index.html:191-193` shows vocab files loading before `game-engine.js`.
- **Suggested fix:** Add a mandatory step in Phase 4 (before editing `index.html`): run `node -c` (or a Node `require`/`vm` parse) on every new part file and abort wiring if any fails.

## Finding 2: Old data files deleted inside Phase 2/3, before Phase 4 wiring exists — guaranteed broken intermediate state across sessions
- **Severity:** Critical
- **Location:** Phase 2, section "Related Code Files" ("Xoa: `js/vocab-data-starters.js`"); Phase 3, section "Related Code Files" ("Xoa: `js/vocab-data-a2key.js`")
- **Flaw:** Phase 2 and Phase 3 each instruct deleting the currently-working monolithic data file as part of that phase, while `index.html` is not updated until Phase 4 (a separate, dependent phase). Phase 3's own risk section calls this "phase quy mo lon nhat" spanning multiple "session/batch" (phase-03 line 56), meaning a session boundary can land exactly between "file deleted" and "index.html rewired."
- **Failure scenario:** Author finishes Phase 2 (deletes `vocab-data-starters.js`, per phase-02 line 30/47) in one session, session ends before Phase 4 runs. `index.html:191` still references `js/vocab-data-starters.js`, which now 404s. `VOCAB_STARTERS` becomes undefined; any invocation of `generateEnglishQuestion()` for grade 1 throws `ReferenceError: VOCAB_STARTERS is not defined` (`js/english-generators.js:53`) whenever a parent picks up work again and opens the app for grade 1 English. The repo is left in a state where English (grade 1) is completely broken, with no documented rollback or "how to resume safely" note anywhere in the plan.
- **Evidence:** phase-02 line 30: "Xoa: `js/vocab-data-starters.js`"; phase-04 line 23: "Modify: `index.html`... them script tag cho tat ca file part MOI TRUOC 2 file aggregator" — confirms index.html wiring is deferred to Phase 4, a separate phase gated on both 2 and 3 completing (plan.md line 45: "Phase 4 phu thuoc Phase 2 VA Phase 3"). No commit/checkpoint guidance exists anywhere in the plan (grepped for "commit|checkpoint|git " across all phase files — zero hits besides an unrelated batch-review mention).
- **Suggested fix:** Do not delete the old monolithic files until Phase 4 completes and verifies the new wiring works; or keep old files present but unreferenced until swap, and only delete in Phase 4's own step as the final action alongside the index.html edit.

## Finding 3: Phase 3's distractor "starving pool" risk is mischaracterized as per-category when it is actually per-POS-across-the-whole-grade — real gap is unverified
- **Severity:** High
- **Location:** Phase 3, Implementation Steps item 7 and Risk Assessment (last bullet)
- **Flaw:** The plan states the risk as "category qua nho (<4-5 tu)... thieu distractor cung `pos`" and treats it as low-priority because "fallback random toan pool da co san." Tracing the actual code shows `generateDistractors` filters candidates from `vocabPool` — the ENTIRE grade-wide array (`VOCAB_STARTERS` or `VOCAB_A2KEY`), not the word's category — so category size is irrelevant to same-POS distractor availability. The real starvation risk is at the POS level across the whole grade pool.
- **Failure scenario:** Neither `vocab-data-starters.js` nor `vocab-data-a2key.js` currently contains a single `pos: "adverb"` entry (verified by grep: only `noun`/`verb`/`adjective`/`number` exist today). Phase 1 explicitly plans to keep adverbs from the Cambridge PDF ("giu lai... vi co gia tri hoc" language only excludes det/pron/conj/prep/poss/dis/int/excl, not adverbs). If Phase 2/3 introduce even a handful of adverb entries, `generateDistractors`'s `samePos` filter (`js/english-generators.js:24-26`) will find zero same-POS candidates for every adverb question and silently fall back to fully random-POS distractors (`js/english-generators.js:44-48`) for every single adverb quiz question — mixing nouns/verbs as wrong answers for an adverb prompt. This is a content-quality defect (POS-mismatched multiple choice), not a crash, so it will never surface as a console error, and Phase 5's spot-check (20-30 random questions) has a low probability of even hitting an adverb question if adverbs are a small fraction of a 1000-word pool.
- **Evidence:** `grep -o 'pos: "[a-z]*"' js/vocab-data-a2key.js | sort | uniq -c` → `21 adjective, 115 noun, 48 verb` (no adverb). `js/english-generators.js:53` `var vocabPool = state.currentGrade === 1 ? VOCAB_STARTERS : VOCAB_A2KEY;` confirms pool is grade-wide, not per-category.
- **Suggested fix:** Add a mandatory Phase 4 step that programmatically counts POS distribution across the final merged pool and flags any POS class with fewer than ~5 members, independent of category size, before sign-off.

## Finding 4: No automated cross-file duplicate check for `en`/`vi`, despite it being a stated Success Criterion
- **Severity:** High
- **Location:** Phase 2 Success Criteria ("Khong trung `en` giua cac file part"); Phase 3 Success Criteria ("Khong trung `en` giua cac file part")
- **Flaw:** Both phases list "no duplicate `en` across part files" as a checkbox success criterion, but no implementation step in Phase 2, 3, or 4 specifies how this is verified — no grep/script step, only "sau moi file doc lai nhanh" (re-read after each file) in Phase 2's mitigation and "doc lai toan bo truoc khi sang batch tiep" in Phase 3's mitigation, both purely manual re-reading across up to 25 files authored over multiple sessions.
- **Failure scenario:** A word like "party" or "look" legitimately appears in more than one Cambridge category/topic list (Appendix 2 topics can overlap, as Phase 1 itself notes needing to de-duplicate categories like "technology" vs "Communication and Technology" — phase-03 line 39). If the same English word is accidentally authored twice across two files (once under an old category retained from the 172-word baseline, once under a new Appendix-2-derived category) with two different `vi` translations, `generateEnglishQuestion` will nondeterministically surface either version depending on `Math.random()` (`js/english-generators.js:63`), meaning the same English word teaches two different Vietnamese meanings across sessions — confusing for a 6-10 year old and undetectable by chance-based manual QA.
- **Evidence:** phase-02 line 48 and phase-03 line 52 list the duplicate-`en` checkbox with no corresponding verification tooling anywhere in Phase 4's "Implementation Steps" (phase-04 lines 26-34 cover only script ordering, const-name collisions, and pos-format spot checks — never a duplicate-`en`/`vi` scan).
- **Suggested fix:** Add a Phase 4 step: concatenate all part files' `en` values (and `vi` values) via a small Node script, and hard-fail if `new Set(list).size !== list.length`.

## Finding 5: Phase 5 manual QA sample cannot realistically bound error rate across 15-25 independently authored files
- **Severity:** High
- **Location:** Phase 5, "Risk Assessment" and Implementation Steps 2-4
- **Flaw:** Phase 5 samples "15-20% tu moi" and "15-20 cau moi loai," described as sufficient because it's "uu tien lay deu tu nhieu category." But the plan itself estimates 15-25 separate part files for A2 Key alone (phase-03 line 18) authored across many sessions/batches (phase-03 line 56), each representing a plausible unit of systematic error (e.g., one session mistranslating an entire batch, or mis-tagging category for one file). A percentage-based word sample does not guarantee per-file coverage: with ~1000 words across 25 files (~40 words/file), a 15-20% random sample (~150-200 words) has a non-trivial chance of drawing zero or only 1-2 words from any single given file, especially if sampling is by category rather than explicitly stratified by source file.
- **Failure scenario:** One entire file (e.g. `vocab-data-a2key-phrasal-verbs.js`, ~40 phrasal verb entries) is authored with a systematic translation drift (e.g. using overly formal/Han-Viet register the plan explicitly warns against in phase-02 line 35) and it is the one file the random sample skips entirely — this whole category ships with lower-quality Vietnamese and is never caught, since there's no acceptance gate requiring at least N words checked per file/category.
- **Evidence:** phase-05 lines 33-34 describe sampling by category, not by file; no minimum-per-file coverage requirement exists in Phase 5's Success Criteria (phase-05 lines 38-44).
- **Suggested fix:** Require Phase 5 to sample at least a fixed minimum count (e.g. 5-8 words) from every part file, not just an aggregate percentage, to guarantee per-file coverage.

## Finding 6: `pos` string format verification is optional/manual, but a silent typo defeats distractor quality with no error signal
- **Severity:** Medium
- **Location:** Phase 4, Implementation Step 7 and Risk Assessment (bullet 3)
- **Flaw:** Step 7 requires checking that `pos` uses full strings (`"noun"`, `"verb"`, etc.) rather than PDF abbreviations, but the verification method offered is "co the viet 1 dong script tam trong console browser" — phrased as optional ("co the"), not a mandatory automated check integrated into the plan's acceptance criteria.
- **Failure scenario:** A typo such as `pos: "adjectiv"` (missing trailing `e`) introduced while hand-transcribing ~700-1000 entries produces no thrown error anywhere — `generateDistractors`'s `w.pos === correctWord.pos` (`js/english-generators.js:25`) simply never matches for that entry, silently degrading to full-random-pool fallback for every question involving that word, and it is invisible in casual UI testing (the word still shows a valid quiz, just with occasionally odd-feeling wrong-POS options, easily mistaken for expected fallback per Finding 3).
- **Evidence:** phase-04 line 34 ("Kiem tra field `pos`...") and line 48 (mitigation says "co the viet 1 dong script tam ... " — conditional phrasing, not a required gate) vs. phase-04 Success Criteria (lines 36-42) which does not list a `pos` value-set check as a checkbox.
- **Suggested fix:** Make the `pos` value audit a mandatory checkbox in Phase 4 Success Criteria: enumerate `new Set(allWords.map(w => w.pos))` and assert it is a subset of exactly `{noun, verb, adjective, adverb, number}` (or whatever the agreed final set is).

## Finding 7: No fidelity check that the 172 baseline translations survive re-transcription into new part files unchanged
- **Severity:** Medium
- **Location:** Phase 2, Implementation Step 3; Phase 1, Implementation Step 7
- **Flaw:** Phase 2 step 3 instructs "giu nguyen ban dich cu" (keep old translation unchanged) for the 172 existing words when copying them into new split files, but the only cross-check specified anywhere is Phase 1's coverage check ("dam bao khong tu nao bi mat" — i.e. checks presence, not translation-text fidelity). There is no automated diff comparing the final `vi`/`pos`/`category` values of the 172 legacy words against their pre-plan values in the current `vocab-data-starters.js`/`vocab-data-a2key.js`.
- **Failure scenario:** During manual re-transcription of ~172 already-verified entries into new grouped files (done by hand/LLM across Phase 2/3 sessions), a translation could be silently altered (e.g. "brave" → "dung cam" accidentally retyped as "dũng cảm" with different diacritics, or a copy-paste shifts one row and pairs the wrong `vi` with an `en`) — a regression in already-shipped, previously-approved content that Phase 5's QA (focused on the ~800+ NEW words, not re-verifying old ones) is not designed to catch.
- **Evidence:** phase-01 line 40 (coverage-only check: "tat ca tu hien co phai xuat hien trong danh sach moi"); phase-02 line 36 (fidelity claim: "giu nguyen ban dich cu (da qua kiem chung...)" with no verification step attached).
- **Suggested fix:** Before deleting the legacy files, generate a diff of `{en, vi, pos, category}` tuples for the 172 baseline words between old and new files, and require it to be empty (or explicitly reviewed) as a Phase 4 gate.

Status: DONE
Summary: Seven evidence-backed findings — two Critical (no syntax-check gate before wiring risks a whole-app outage since game-engine.js drives both subjects and loads after vocab files; deleting old data files mid-phase before Phase 4 rewires index.html leaves a guaranteed broken intermediate state across sessions), three High (distractor-starvation risk misdiagnosed as category-level instead of the actual grade-wide POS-level gap with zero adverb entries today; no automated duplicate-en/vi check despite being a stated success criterion; Phase 5's percentage-based sampling doesn't guarantee per-file coverage across 15-25 independently authored files), two Medium (pos-value audit is optional not mandatory; no fidelity diff protecting the 172 already-verified legacy translations during re-transcription).
Concerns/Blockers: None beyond the findings above.
