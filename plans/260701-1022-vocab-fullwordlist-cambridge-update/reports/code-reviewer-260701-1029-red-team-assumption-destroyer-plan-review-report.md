# Red-Team Review: 260701-1022-vocab-fullwordlist-cambridge-update

Reviewer role: Scope Auditor / Assumption Destroyer. Verified against actual cached PDF extraction (`scratchpad/vocab-pdfs/starters.txt`, `a2key.txt`), repo code (`js/english-generators.js`, `js/vocab-data-a2key.js`), and prior research report.

## Finding 1: Appendix 2 topic-list columns have no delimiter in extracted text — Phase 1 tokenization is unspecified and will silently corrupt multi-word terms
- **Severity:** Critical
- **Location:** Phase 1, "Implementation Steps" step 5 (Appendix 2 parsing); referenced as the "official category source" throughout Phase 1/3
- **Flaw:** The plan assumes Appendix 2 topic lists can be parsed to assign categories ("Doi chieu voi danh sach A-Z ... de gan category") but never specifies how to split the 4-column rows into individual entries. The actual pypdf extraction merges all 4 columns onto one whitespace-separated line with no column boundary marker.
- **Failure scenario:** Real extracted line (`a2key.txt:2521`): `bag get dressed raincoat tie` — four columns: "bag" | "get dressed" | "raincoat" | "tie". A naive split-on-whitespace parser (which is the only approach the plan describes) yields 5 tokens: "bag", "get", "dressed", "raincoat", "tie" — silently fabricating two bogus single-word entries ("get", "dressed") instead of the real multi-word term "get dressed". Same problem recurs for "washing machine" (`a2key.txt:2514`), "mobile (phone)" (`a2key.txt:2512`), "swimming costume" (`a2key.txt:2533`), "cell phone" (`a2key.txt:2512`) — all appear in the same space-delimited rows as single-word neighbors, so token count per row is unpredictable and column alignment is destroyed by pypdf's text-stream extraction.
- **Evidence:** `a2key.txt:2510-2533` (Appliances/Clothes and Accessories/Colours tables), Phase 1 step 5 wording gives zero guidance on tokenizing rows before matching.
- **Suggested fix:** Add an explicit sub-step: reconstruct columns using x-coordinate/layout-aware extraction (pypdf's `extract_text(extraction_mode="layout")` or `pdfplumber` word-position based grouping) rather than plain `extract_text()`, or manually verify every topic-list row against the source PDF's visual columns before trusting the split. Do not rely on whitespace splitting.

## Finding 2: Plan self-contradicts on whether translation is in-scope for this plan
- **Severity:** Critical
- **Location:** `plan.md` "Overview rui ro chinh" (line 30) vs Phase 2 and Phase 3 titles/steps
- **Flaw:** `plan.md:30` states explicitly: "Ban dich se duoc tac gia hoa thu cong/qua LLM trong buoc `/ck:cook`, khong phai trong plan nay — plan chi dinh nghia cau truc, nguon du lieu chinh xac, va tieu chi QA" (translation happens later in `/ck:cook`, NOT in this plan). Yet Phase 2 is titled "Bien dich Starters day du" and Phase 3 "Bien dich A2 Key day du" — both contain literal translation implementation steps (phase-02.md steps 2,3; phase-03.md steps 2,5) and success criteria that judge translation quality directly ("Moi ban dich tieng Viet tu nhien, dung nghia...").
- **Failure scenario:** Whoever executes this plan (agent or human) cannot tell whether Phase 2/3 should actually produce translated Vietnamese text now, or just produce the extracted/structured English word list and defer translation to a separate `/ck:cook` invocation. If the executor follows the Overview literally and skips translation, Phase 2/3 success criteria (which require Vietnamese translations) cannot be met. If the executor follows the phase files literally, the plan has quietly absorbed the 700-1000+ word manual translation effort the Overview claims is out of scope, with no separate review/approval checkpoint for that much larger scope.
- **Evidence:** `plan.md:30`; `phase-02-bien-dich-starters-day-du.md:35-40` (steps 2-3, translation instructions); `phase-03-bien-dich-a2-key-day-du.md:38,41` (steps 2, 5, translation instructions); Success Criteria in both phase files reference "ban dich tu nhien, dung nghia".
- **Suggested fix:** Resolve the contradiction explicitly — either strip translation instructions out of Phase 2/3 (leaving only structuring/English-side work, deferring `vi` field population to `/ck:cook`) or correct the Overview to state translation IS in scope of this plan and drop the `/ck:cook` deferral claim.

## Finding 3: Multi-line-wrapped parenthetical annotations in the thematic list will break naive text matching
- **Severity:** High
- **Location:** Phase 1, Implementation Steps 3 (thematic list cross-reference) and Risk Assessment item 2
- **Flaw:** The plan assumes cross-referencing A-Z wordlist entries (which include disambiguating annotations like `bat (as sports equipment) n`) against the thematic-list word clusters will correctly assign category and grade level. It does not account for pypdf line-wrapping splitting these annotations across two lines.
- **Failure scenario:** Real extraction shows `bat (as sports \nequipment)` split across two lines (`starters.txt:6379-6380`) and `favourite \n(US favorite)` similarly split (`starters.txt:6393-6394`). A line-oriented regex parser (the only method described in the plan) reading one line at a time will either treat "bat (as sports" as a malformed/dropped entry, or need extra multi-line-join logic the plan never mentions. This directly threatens Phase 1's own risk mitigation, which assumes matching disambiguated forms like "bat (as sports equipment)" will reliably distinguish it from the homonym "bat" in the Animals topic.
- **Evidence:** `starters.txt:6379-6380`, `starters.txt:6393-6394`; Phase 1 "Implementation Steps" step 3 and "Risk Assessment" bullet 2 describe only single-line pattern matching (`regex tren pattern 'tu + tag ngu phap'`).
- **Suggested fix:** Add explicit line-joining/normalization step before regex matching (e.g., join lines where a line ends without a POS tag and the next line starts with `(` or lowercase continuation), and spot-check specifically for wrapped multi-word/annotated entries, not just random samples.

## Finding 4: Compound POS tags in A2 Key alphabetic list break the "exclude by pos" filtering rule
- **Severity:** High
- **Location:** Phase 1, Implementation Steps step 4
- **Flaw:** Step 4 says to exclude grammar-only entries by checking pos against `det/pron/conj/prep/...`, implying entries have a single, unambiguous POS tag. Real A2 Key data has compound tags combining function-word and content-word senses on one entry.
- **Failure scenario:** `a2key.txt:130-136` show entries like `a/an (det)`, `a few (det, adj & pron)`, `about (adv & prep)`, `above (adv & prep)`, `across (adv & prep)`. A pure exclude-if-contains-det/pron/prep rule would drop `a few` — a genuinely useful quantity phrase for a 9-10 year old — while a pure include-if-contains-adj/adv rule would keep every "adv & prep" function word (about, above, across, after, again, against...) as if it were content vocabulary, defeating the stated goal of excluding grammar words. The plan provides no resolution rule for entries carrying both types of tags simultaneously.
- **Evidence:** `a2key.txt:130-174` (multiple compound-tag entries); Phase 1 step 4 wording: "Loai tu ngu phap tuong tu buoc 2 (det/pron/conj/prep/av/mv thuan chuc nang...)".
- **Suggested fix:** Define an explicit tie-break rule for compound tags (e.g., keep only if ANY tag is a content POS AND the word has a concrete, teachable meaning; maintain an explicit allow/deny list for the ~10-20 ambiguous compound-tag entries rather than a blanket rule) and verify the list against the real compound-tag entries before Phase 2/3 begin.

## Finding 5: Distractor generator dedups by Vietnamese translation — already causing collisions today, will worsen sharply at 5-8x volume, and no plan step audits it
- **Severity:** High
- **Location:** Phase 3 Risk Assessment (bullet on distractor fallback), Phase 4 Success Criteria/Risk Assessment
- **Flaw:** `generateDistractors` in `js/english-generators.js:29-40` excludes candidate distractors if either their `en` OR their `vi` value has already been used (`usedVi[correctWord.vi] = true` at line 32, checked at line 36). This means two English words sharing one Vietnamese translation can never both appear as options — reducing the effective distractor pool independently of the `pos`-based filter the plan repeatedly cites as the only capacity risk.
- **Failure scenario:** This collision already exists in production data: `js/vocab-data-a2key.js:4` ("wake up" → "thức dậy") and `:14` ("get up" → "thức dậy") share a translation. At 172 words this is rare; at 700-1000+ words with near-synonym pairs translated independently across many batches by different translation passes (as Phase 3 explicitly plans, "chia thanh session/batch"), duplicate `vi` values across unrelated `en` words (e.g. "big"/"large", "look"/"watch", "happy"/"glad") become statistically likely and will silently shrink the pool below what Phase 3's own capacity check (only counting same-`pos` word count per category) accounts for. None of the phases include a cross-check for duplicate `vi` values across the full expanded pool.
- **Evidence:** `js/english-generators.js:23-50` (full dedup logic); `js/vocab-data-a2key.js:4,14` (existing real-world collision); Phase 3 line 43 and Phase 4 line 48 only discuss `pos`-based distractor scarcity, never `vi`-value collisions.
- **Suggested fix:** Add a Phase 4 verification step that scans the aggregated `VOCAB_STARTERS`/`VOCAB_A2KEY` arrays for duplicate `vi` strings and flags categories where duplicate-`vi` density could starve `MC_EN_VN`/`LISTEN_CHOOSE` distractor generation (both modes display `.vi` as the visible option set).

## Finding 6: Translation-quality success criteria are unfalsifiable given the review method described
- **Severity:** Medium
- **Location:** `plan.md:54`, Phase 2 Success Criteria, Phase 3 Success Criteria, Phase 5 step 4
- **Flaw:** Success criteria such as "Moi ban dich tieng Viet chinh xac, tu nhien, phu hop tre em" (plan.md:54) have no objective measurement. The only verification method specified is the same actor re-reading their own translation ("sau moi file doc lai nhanh 100%" — Phase 2 risk mitigation; "doc lai toan bo truoc khi sang batch tiep" — Phase 3 risk mitigation) plus a Phase 5 QA pass sampling only 15-20% of the 700-1000+ new A2 Key words.
- **Failure scenario:** A translator (human or LLM) with a systematic blind spot (e.g., consistently mistranslating a specific idiom pattern, or using unnatural word order) will pass their own self-review because the review method is "read it again," not an independent check. Phase 5's 15-20% sample size on ~1000 words still leaves 800+ words unverified, and there is no rubric for what "loi dich ro rang" (obvious translation error) means operationally.
- **Evidence:** `plan.md:54`; `phase-02-bien-dich-starters-day-du.md:52` (risk mitigation is self-review only); `phase-03-bien-dich-a2-key-day-du.md:56` (same); `phase-05-qa-thu-cong-tren-trinh-duyet.md:33,48` (15-20% sample, no rubric).
- **Suggested fix:** Either explicitly accept this quality risk as a known limitation for `/ck:cook` follow-up (consistent with Finding 2's resolution), or define a concrete rubric/second-pass reviewer requirement before marking Phase 2/3/5 success criteria complete.

## Finding 7: Grade-boundary (Starters vs Movers/Flyers) segmentation relies on a flat, unsegmented word run with only a 20-30 word spot-check as verification
- **Severity:** Medium
- **Location:** Phase 1 Risk Assessment bullet 2; Implementation Steps step 3
- **Flaw:** The thematic-list extraction interleaves all three grade columns (Starters/Movers/Flyers) into one flat sequential run per topic with no column markers, confirmed by the real extraction. The plan's stated mitigation — "chi giu tu co mat trong A-Z wordlist chinh thuc" — correctly avoids needing to know the column boundary for grade-inclusion, but the plan still asserts (step 3) that you can identify "khoi dau tien" (the first block) per topic before the boundary shifts to Movers/Flyers, which is not reliably locatable in the flat text.
- **Failure scenario:** For the "Animals" topic, `starters.txt:5697-5757` shows ~60 words with no visual/textual separator between the Starters segment, the Movers segment, and the Flyers segment (e.g. "zoo" at line 5727 is followed immediately by "bat" at line 5728, which visually starts the Movers column in the source PDF table but appears as a plain continuation in extracted text). If category-assignment logic ever needs to reason about "the Starters block" directly (rather than only doing pure membership lookup against the A-Z list), it will silently misattribute category for words near a grade boundary.
- **Evidence:** `starters.txt:5692-5790`; Phase 1 step 3 wording ("khoi dau tien truoc khi chuyen sang tu vung Movers/Flyers... "); Risk Assessment bullet 2's mitigation is sound only if strictly limited to "lookup, never positional reasoning" — the plan text does not state that constraint explicitly, leaving room for an implementer to use positional heuristics that will fail.
- **Suggested fix:** Make explicit in Phase 1 that category assignment must use set-membership lookup only (word is in A-Z Starters list AND word appears anywhere in the topic's flat text = category match), never positional/ordinal reasoning about "first N words" per topic.
